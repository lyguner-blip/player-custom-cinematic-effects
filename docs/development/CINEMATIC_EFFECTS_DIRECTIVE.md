# 电影级特效库后续开发指令

本文档是 `player-custom-cinematic-effects` 后续开发的长期约束。后续 Codex 或协作者处理本 MOD 时，必须优先遵守这里的产品边界、结构约定和验收标准。

## 一、产品定位

本 MOD 只负责 Foundry VTT 的电影级视觉与音效演出。它不是轻量提示库，也不是自动化规则库。

- 取消轻量、标准等特效质量等级，只保留电影级演出规格。
- 所有效果默认按电影级制作：多层素材、镜头辅助、音效、Token 动势、浮字和必要的持续/清理逻辑要作为完整体验考虑。
- 如果资源库中缺少足够好的素材，或现有效果明显达不到电影级，必须第一时间在更新说明和最终回复里向维护者反馈。

## 一点五、规则与参考资源

基础特效创作时，必须以规则书 MOD 提供的法术描述为准：

- 规则书源路径：`D:\fvtt\Data\modules\dnd-players-handbook`
- 参考动画源路径：`D:\fvtt\Data\modules\dnd5e-animations`

约束如下：

- `dnd-players-handbook` 是法术语义与表现的唯一来源，优先级最高。
- `dnd5e-animations` 只作为预设与素材编排参考，不得直接替代规则描述。
- 当参考动画和规则书描述不一致时，以规则书为准。
- 若某个效果无法从现有资源中还原描述，应在更新说明里明确写出缺口。

## 二、基础通用特效库

后续要建立一个不绑定任何角色的基础特效库。它负责为通用物品、通用法术和通用动作提供可复用的电影级效果。

- 基础库不得依赖角色名、actor id 或角色档案。
- 第一阶段先完成所有法术戏法的通用特效。
- 第二阶段扩展到所有常用法术。
- 第三阶段覆盖武器、消耗品、职业通用动作和环境交互。
- 基础库应优先沉淀通用构件：施法前摇、投射物、命中、地面符文、持续光环、解除/清理、音效、浮字。

## 三、定制特效与通用特效的关系

定制特效是对通用特效的延展和提升，而不是完全重写另一套系统。

- 每个定制效果应能复用基础特效的核心阶段，再叠加角色主题层。
- 角色主题层可以修改颜色、材质、额外素材、浮字、音效、镜头节奏和多段触发。
- 例如火元素血统角色施放通用法术时，应在通用法术效果上叠加火焰、热浪、熔光或火星层，而不是复制一份不可维护的完整动画。
- 通用效果必须暴露清晰的扩展点，至少包括 `cast`、`projectile`、`impact`、`area`、`sustain`、`cleanup`、`sound`、`floatingText`。

## 四、玩家切换

玩家必须能在特效面板中自由切换定制特效和通用特效。

- 当某个 item 存在角色定制特效时，默认使用定制特效。
- 玩家可以切到通用特效进行对比或回退。
- 玩家也可以关闭本模块对该 item 的演出。
- 面板上的切换必须清晰，不能让 DM 需要逐条解释。

## 五、自动化接入

本 MOD 要为 `player-custom-automation-effects` 等自动化 MOD 提供稳定入口。

- 自动化 MOD 应能按阶段触发动画，例如移动前、移动后、攻击前、命中后、伤害后、效果创建后、效果删除后。
- 特效播放 API 要支持传入上下文：actor、token、item、activity、workflow、targets、templates、damage、attackRoll、自定义 stage。
- 多段动画必须能复用同一个效果定义，不要要求自动化 MOD 硬编码每一段 Sequencer 细节。
- 任何自动化接入新增 API，都要在 README 或开发文档中说明调用方式。

## 六、外部动画优先级

本 MOD 是玩家定制电影特效的最高级入口。

- 当定制库或通用基础库存在某个 item 的动画时，Boss Loot FX、Auto Animations 等其他动画 MOD 默认不再触发同一 item 的动画。
- 禁止双重播放造成画面污染。
- 如果外部 MOD 同时负责规则自动化，必须只关闭外部动画，不要破坏规则效果。
- 外部动画状态要在管理面板中用简洁标识呈现。

## 七、角色卡标记

对玩家而言，如果某个 item 存在本 MOD 的定制特效或通用特效，角色卡中应有轻量明确的标记。

- 标记要能区分：定制特效、通用特效、已关闭、被外部动画保护。
- 标记不应破坏 DnD5e 默认角色卡布局。
- 鼠标悬停应显示简短说明。

## 八、管理面板

管理面板目标是减轻 DM 压力，而不是增加配置负担。

- 首屏只显示最关键状态：角色、模式、可自动数量、外部动画冲突、最近触发。
- 高级配置折叠，避免面板过长。
- 批量操作要优先支持：按角色启用/关闭、按模式切换、清理冲突、扫描缺失资源。
- 面板文案要短，操作控件要明确。

## 九、更新说明规范

每次修改仓库后，必须在 `docs/updates/` 下新增一份更新说明。

文件命名格式：

```text
YYYY-MM-DD-HHmmss-简短主题.md
```

每份更新说明至少包含：

- 更新时间，使用中国时区。
- 本次目标。
- 已完成改动。
- 验证方式。
- 已知风险或后续待办。
- 如有资源缺失或效果欠佳，必须明确列出。

## 十、后续推荐结构

后续实现基础通用特效库时，建议按以下结构推进：

```text
scripts/
  universal/
    cantrips.mjs
    spells-level-1.mjs
    effect-builder.mjs
    registry.mjs
  profiles/
    <character>.mjs
docs/
  development/
  updates/
```

`scripts/universal/` 负责通用电影级效果。`scripts/profiles/` 只负责角色定制覆盖与扩展。两个层级不得互相污染。

## 十一、验收标准

每次相关开发完成前至少确认：

- `node --check scripts/main.mjs` 通过。
- 新增或修改的 `.mjs` 文件语法检查通过。
- 面板能明确看到电影级定位，不再出现轻量/标准档位。
- 新增通用效果不依赖角色档案。
- 定制效果优先级高于通用效果，通用效果优先级高于外部动画。
- 更新说明已写入 `docs/updates/`。
